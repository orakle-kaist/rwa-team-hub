# 03 — 토큰화 · 커스터디

---

## 1. 두 질문, 두 표준

규제와 결제 현실을 코드로 옮기면 두 질문으로 압축된다.

| 질문 | 표준 | 담당 |
|------|------|------|
| **누가** 보유·이전할 수 있는가 | **ERC-3643** | 매 이전마다 신원·규칙 검사, 동결·강제이전 등 법적 기능 |
| 발행·환매가 **언제** 확정되는가 | **ERC-7540** | 1차 시장의 실물 결제 대기·되돌림 상태 표현 |

두 표준은 경쟁이 아니라 곱셈이다. 하나는 규제를, 하나는 시차를 맡는다.

---

## 2. 왜 검사가 토큰에 내장돼야 하는가

| 방식 | 확인 시점 | 문제 |
|------|-----------|------|
| 전통 증권 시장 | 계좌 개설 시 1회 | 이후 거래가 증권사 시스템 안에 갇혀 있어 통제 유지 |
| 일반 토큰 (ERC-20) | 발행 시 1회 | 이후 익명 지갑으로 자유 이전 → **통제 지점이 사라짐** |
| **ERC-3643** | **모든 이전마다** | 검사가 토큰 자체에 내장 |

ERC-20의 `transfer()`가 잔액만 확인하는 것과 달리, ERC-3643은 이전 성립 전에
`canTransfer()`를 호출해 수신자 신원 클레임을 확인하고 Modular Compliance 모듈을 순차 실행한다.
하나라도 통과하지 못하면 거래는 **revert**된다.

> 위반이 사후 적발되는 게 아니라, **위반이 되는 거래 자체가 성립하지 않는다.**

## 3. Modular Compliance — 5개 모듈

한국 주식 RWA는 판정 항목이 5개이고 참조하는 데이터 소스와 판정 단위가 서로 다르므로
단일 화이트리스트로 묶을 수 없다. 모듈로 분리한다.

| 순서 | 모듈 | 검사 내용 | 데이터 소스 | 판정 단위 | 연산 비용 |
|------|------|-----------|-------------|-----------|-----------|
| 1 | 신원·자격 | 필수 클레임(KYC·적격성·관할·조세 거주성)을 유효한 issuer 서명으로 보유하는가 | ONCHAINID 클레임 | 개인 | 낮음 |
| 2 | 락업 | 보호예수·동결 수량에 해당하지 않는가 | 토큰 내부 상태 | 개인 | 낮음 |
| 3 | 적격 보유 화이트리스트 | 수신 주소가 커스터디 구조상 보유 가능한 적격 주소인가 | 어댑터가 갱신하는 온체인 매핑 | 주소 | 낮음 |
| 4 | 제재 스크리닝 | 송신자·수신자가 제재 대상인가 | 클레임 또는 온체인 레지스트리 | 개인 | 중간 |
| 5 | **외국인 보유한도** | 이전 후 외국인 총보유 비율이 법정 한도를 넘지 않는가 | **오라클 공급 외부 데이터** | **종목 전체 집계** | 높음 |

### 검사 순서는 가스 효율 기준

법령이 정한 우선순위가 아니라 설계 선택이다. 단순 조회로 판별 가능한 검사를 앞에 두면
실패 시 소모되는 가스를 줄일 수 있다.

### 5번 모듈은 성격이 다르다

1~4번은 이미 온체인에 존재하는 상태를 조회하지만, 5번은 온체인에서 **생성할 수 없는 값**을
외부에서 공급받아야 한다. → [02-infrastructure.md](./02-infrastructure.md) §오라클

### revert reason 규칙 (필수)

각 모듈의 실패는 **어느 관문에서 차단됐는지 구분 가능한** revert reason을 반환해야 한다.
구분할 수 없으면 감사와 운영 대응이 불가능하다.

특히 외국인 보유한도 모듈은 아래를 남겨야 사후에 차단의 정당성을 검증할 수 있다.
- 종목 코드
- 적용된 법정 한도
- 오라클 기준 시각(`updatedAt`)
- 이전 후 예상 보유 비율

## 4. mint 경로의 컴플라이언스 갭 (알려진 결함 — 반드시 대응)

ERC-3643의 기본 `mint` 경로는 `from`이 zero address이므로 **전송이 아니라 발행으로 처리**되어,
`IdentityRegistry.isVerified(to)`만 확인한 뒤 **Modular Compliance 모듈을 건너뛴다.**

즉 발행 시점에는 락업·화이트리스트·제재·외국인 한도 검사가 적용되지 않는다.
외국인 투자자에게 토큰이 처음 지급되는 경로가 바로 mint이므로, 이건 중대한 위험이다.

**대응:** `moduleMintAction` 훅(또는 동등한 수단)을 적용해 발행 경로에도 동일한 모듈 검사를 강제한다.
`moduleMintAction`은 `mint()` 내부에서 토큰 발행 **전에** 호출된다. `IComplianceModule.transferred()`와 동일한 책임 구조이지만 `from`이 zero address이므로 5개 모듈이 모두 zero address를 처리할 수 있어야 한다.
→ `AGENTS.md` R6

> 발행 흐름 통제(§6의 Pending→Claimable)만으로는 부족하다. 그건 운영 절차이지 컨트랙트 레벨 강제가 아니다.
> 둘 다 필요하다.

## 5. 미리 수탁 (Pre-custody) — 1차와 2차의 분리

**전제:** 발행회사가 발행인계좌관리기관으로 참여하기로 이미 정해져 있으므로, 대상 주식은
**미리 수탁**되어 토큰을 뒷받침한다.

| | 2차 시장 | 1차 시장 |
|---|----------|----------|
| 무엇 | 토큰 매수·매도 (투자자 ↔ 투자자) | 발행(creation)·환매(redemption) |
| 빈도 | **대부분의 거래** | 가끔 (순 수탁량이 바뀔 때만) |
| 소요 | **즉시** | 2~4 영업일 |
| 이유 | 미리 수탁된 주식이 뒷받침 → 원장에서 소유자만 변경 | 실물 매수·매도 + 환전 필요 |
| 대기 상태 | **없음** | 있음 (ERC-7540) |
| 검사 | 이전마다 신원·한도 검사 후 즉시 확정 | 발행 전 동일 검사 + 실물 확보 확인 |

> **구현 규칙:** 2차 경로에 request/claim 비동기 패턴을 넣지 말 것.
> 대기는 1차에만 존재한다. → `AGENTS.md` R4

## 6. 1차 시장 비동기 흐름 (ERC-7540)

투자자가 대금을 낸 시점과 수탁기관이 실물을 확보해 계좌부에 반영되는 시점 사이에는 시차가 있다.
대금 입금 즉시 토큰을 발행하면 실물 뒷받침 없는 토큰이 생긴다.

### 발행 (Creation)

```
requestDeposit()          대금 입금, 요청은 Pending
      │
      │  ← 오프체인: 원화 환전 · 실물 매수 체결 · 계좌대체 (T+2)
      ▼
custody attestation       어댑터가 실물 확보를 확인하고 적격 보유 화이트리스트 등재
      │
      ▼
Claimable                 이 시점부터 수령 가능
      │
   claim()                컴플라이언스 검사 통과 후 토큰 발행
      ▼
Minted
```

- Pending 구간의 길이를 결정하는 것은 블록 생성 속도가 아니라 **오프체인 절차 3개**
  (외국환 자금 유출입·원화 환전, KRX 매수 체결, 전자등록계좌부 계좌대체)다.
- 온체인이 할 일은 이 시차를 없애는 게 아니라, **시차가 존재하는 동안의 상태를 명시적으로 관리**해
  어느 시점에 무엇이 확정됐는지 분명히 하는 것이다.

### 환매 (Redemption)

```
requestRedeem()   토큰 잠금, 요청은 Pending
      │
      │  ← 오프체인: 실물 매도 · 재환전 (T+2)
      ▼
fulfillRedeem()   토큰 소각 (빠져나가는 경로이므로 적격성 검사 미적용)
      │
      ▼
withdraw()        대금 반환
```

### 실패 경로 (필수 구현)

실물 매수 실패, 환전 실패, 컴플라이언스 재검사 실패 시 **되돌아갈 경로**를 남긴다.
`void` / `refund` / `manualReview` 중 하나로 분기하며, 조용히 삼키지 않는다.

## 7. 배당 처리

- 배당은 **투자자별로 따로 계산**한다. 주가(NAV)에 섞지 않는다.
- 이유: 투자자의 거주국·조세조약에 따라 원천징수율이 다르므로, 섞으면 개별 세율을 반영할 수 없다.
- 조세 거주성은 ONCHAINID 클레임에서 읽는다.
- 배당락·배당률은 오라클(기업행위 피드)이 공급한다.

> 예시 수치(15% 등)는 **예시일 뿐이다.** 실제 세율은 투자자 거주국과 조약에 따라 다르므로
> 하드코딩하지 말고 클레임 기반으로 조회한다.

## 8. 구현 체크리스트

- [ ] 5개 컴플라이언스 모듈이 각각 독립 모듈로 구현됐다
- [ ] 각 모듈이 구분 가능한 revert reason을 반환한다
- [ ] 외국인 한도 모듈이 오라클 값을 쓰고, 스테일 데이터를 보수적으로 거절한다
- [ ] mint 경로에도 모듈 검사가 강제된다 (`moduleMintAction` 또는 동등)
- [ ] 2차 매매가 단일 트랜잭션에서 동기 완결된다 (대기 상태 없음)
- [ ] 1차 발행이 Pending → Claimable → Minted 상태를 거친다
- [ ] 1차 실패 경로(void/refund/manualReview)가 구현·테스트됐다
- [ ] 배당이 투자자별 세율로 계산된다 (하드코딩 아님)
