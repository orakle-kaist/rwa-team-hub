# 20 — 트랙별 구현·통합 플레이북

> 대상: 개발 경험이 많지 않은 트랙 팀원, 구현 담당자, AI 코딩 에이전트, 통합 담당자
>
> 이 문서는 **일하는 방법**을 설명한다. 제품 규칙의 단일 진실 공급원은 여전히 00~13 문서이며,
> 내용이 충돌하면 이 문서가 아니라 원래 명세를 따른다.

---

## 1. 이 문서를 어디에 두는 이유

상세 안내서는 `docs/`에 두고, 루트 `README.md`에는 이 문서로 오는 짧은 링크만 둔다.

- `README.md`는 처음 들어온 사람이 프로젝트와 시작점을 빠르게 찾는 **현관**이다.
- 이 문서는 트랙별 책임, 작업 순서, 테스트, 인계 방법을 담는 **작업 설명서**다.
- 01~04 문서는 각 트랙이 지켜야 할 제품·기술 규칙을 담는 **설계도**다.
- [11-interfaces.md](./11-interfaces.md)는 트랙 사이를 연결하는 **플러그 규격**이다.
- [12-acceptance.md](./12-acceptance.md)는 구현 완료를 판정하는 **검수표**다.
- [13-open-decisions.md](./13-open-decisions.md)는 팀원이 임의로 정하면 안 되는 **보류 결정 목록**이다.

같은 내용을 README와 트랙 문서에 복사하지 않는다. 중복된 설명은 시간이 지나면 서로 달라진다.

코드가 생긴 뒤 `contracts/`나 각 `services/` 아래에 둘 로컬 README에는 **설치·실행·설정·디버깅 방법만** 적고,
트랙의 목표와 규칙은 이 문서와 01~04 문서로 링크한다. 작업 카드와 PR 양식이 실제로 몇 차례 검증된 뒤에는
`.github` 이슈·PR 템플릿으로 옮겨 반복 작업을 줄일 수 있지만, 별도의 트랙 설명서를 여러 벌 만들지는 않는다.

## 2. 네 트랙을 한눈에 보기

네 트랙은 네 개의 독립 제품이 아니라 앞뒤가 맞물리는 한 시스템이다.

| 트랙 | 쉬운 질문 | 주된 구현 결과 | 다음 트랙에 넘기는 것 |
|------|-----------|----------------|------------------------|
| ① 규제 | 누가 이 주식을 보유해도 되는가? | 신원·클레임 검증, 안전한 테스트 신원 | 검증 함수, 클레임 상태, 실패 사례 |
| ② 인프라 | 외부 데이터와 두 원장을 어떻게 안전하게 잇는가? | 오라클, 커스터디 모의 어댑터, 1차 DvP | 피드·attestation·commit/abort 인터페이스 |
| ③ 토큰화 | 주식의 발행·이전·배당·환매를 어떻게 코드로 강제하는가? | 주식 토큰, 5개 규제 모듈, 1차 볼트, 배당 | ABI, 이벤트, 오류, 배포 주소 |
| ④ 에이전트 | 사용자의 말을 어떻게 안전한 주문으로 바꾸는가? | 의도 파싱, 위임 검사, 주문 실행·설명 | 사용자용 실행 결과와 거절 설명 |

현재 디렉터리 계획상 주 작업 위치는 다음과 같다. 아직 없는 폴더는 0단계 스캐폴딩 PR에서 만든다.

| 트랙 | 주 작업 위치 |
|------|--------------|
| ① 규제 | `contracts/src/identity/`, `contracts/test/` |
| ② 인프라 | `contracts/src/oracle/`, `services/custody-adapter/`, `services/orchestrator/` |
| ③ 토큰화 | `contracts/src/token/`, `contracts/src/primary/`, `contracts/test/` |
| ④ 에이전트 | `services/agent/` |

### 꼭 한 명 정할 역할: 통합 담당자

각 트랙 담당과 별도로 **통합 담당자 한 명**을 정한다. 새로운 시스템 액터가 아니라 팀 운영 역할이다.

통합 담당자는 다음을 맡는다.

- [11-interfaces.md](./11-interfaces.md)의 변경을 한곳에서 확인한다.
- 트랙이 제공한 ABI, 타입, 모의 데이터, 실행 방법이 서로 맞는지 확인한다.
- 매주 작은 통합 PR을 만들고 AC-10 전체 흐름을 계속 연결한다.
- 미결정 사항을 발견하면 [13-open-decisions.md](./13-open-decisions.md)에 올린다.

> 트랙별 코드를 몇 주 동안 따로 만든 뒤 마지막 날 한꺼번에 합치지 않는다.
> 인터페이스와 모의 구현부터 일찍 합치고, 작동 범위를 매주 넓힌다.

## 3. 개발자가 아니어도 맡을 수 있는 일

모든 작업은 가능하면 **도메인 담당자 1명 + 구현 담당자 1명**이 짝을 이룬다. 구현 담당자는 사람 개발자일 수도,
AI 코딩 에이전트를 사용하는 팀원일 수도 있다.

비개발 팀원이 맡을 수 있는 핵심 역할은 다음과 같다.

- 명세에 적힌 성공·실패 시나리오가 실제 업무 흐름과 맞는지 설명한다.
- 테스트가 통과했다는 말만 듣지 않고, 입력·결과·오류 메시지를 직접 확인한다.
- 개인정보, 법률 판단, 미결정 파라미터가 코드에 몰래 들어가지 않았는지 검수한다.
- 모의 구현의 가정과 실제 연계 시 달라질 점을 기록한다.
- 데모 순서와 사용자 설명 문구를 다듬는다.

알아두면 좋은 네 단어는 이것뿐이다.

| 단어 | 뜻 |
|------|----|
| 인터페이스 | 다른 트랙이 호출할 수 있도록 약속한 함수와 데이터 모양 |
| 모의 구현(Mock) | 실제 KSD·KRX·은행 대신 성공·지연·실패를 재현하는 시뮬레이터 |
| 테스트 Fixture | Alice, Bob, SK하이닉스처럼 모든 팀이 반복해서 쓰는 가짜 입력 묶음 |
| 인수 테스트 | “구현이 끝났다”를 말이 아니라 실행 결과로 증명하는 검사 |

## 3a. 트랙별 RACI (역할 분류)

> **R** Responsible — 직접 만들거나 결정하는 사람  
> **A** Accountable — 결과에 서명하는 사람 (R에 포함 가능)  
> **C** Consulted — 만들기 전에 의견을 구하는 사람  
> **I** Informed — 완료 후 결과를 공유받는 사람

| 활동 | 트랙 ① 규제 | 트랙 ② 인프라 | 트랙 ③ 토큰화 | 트랙 ④ 에이전트 | 통합 담당자 |
|------|:-----------:|:------------:|:------------:|:--------------:|:-----------:|
| 클레임 topic 번호 확정 | **R/A** | I | C | I | C |
| 오라클 mock 기본값 설정 | I | **R/A** | C | I | C |
| 컴플라이언스 모듈 인터페이스 확정 | C | C | **R/A** | I | **A** |
| 에이전트↔토큰 연결 인터페이스 | I | I | C | **R/A** | **A** |
| 11-interfaces.md 변경 | C | C | C | C | **R/A** |
| 미결정 항목(D-B6~D-B9) 결정 촉구 | C | C | C | C | **R/A** |
| mock ABI 호환성 검증 스크립트 | I | I | I | I | **R/A** |
| 통합 데모 시나리오 준비 | C | C | C | C | **R/A** |

## 4. 모든 팀이 작업을 시작하는 같은 방법

### 4.1 읽는 순서

작업마다 다음 순서를 지킨다.

1. 루트 [AGENTS.md](../AGENTS.md)를 읽는다.
2. [00-cycle-overview.md](./00-cycle-overview.md)에서 전체 흐름을 확인한다.
3. 담당 트랙 문서(01~04)를 읽는다.
4. [10-domain-model.md](./10-domain-model.md)의 INV-1~INV-9를 확인한다.
5. [11-interfaces.md](./11-interfaces.md)에서 자신이 만들거나 호출할 인터페이스를 확인한다.
6. [12-acceptance.md](./12-acceptance.md)에서 맡을 AC 번호를 고른다.
7. [13-open-decisions.md](./13-open-decisions.md)에 관련 미결정 사항이 있는지 확인한다.

### 4.2 작업 크기

“트랙 전체 구현”처럼 큰 작업을 한 번에 요청하지 않는다. **AC 한 개 또는 서로 강하게 묶인 AC 몇 개**를
한 작업 단위로 삼는다. 예를 들어 “오라클 구현”보다 “AC-3.4: 오래된 오라클 값은 거절”이 좋은 작업명이다.

### 4.3 작업 카드 양식

이 양식을 이슈, 작업 요청, AI 코딩 에이전트 프롬프트에 그대로 사용한다.

```text
작업명:
책임 트랙:
책임 AC:
반드시 읽을 문서/절:
받는 입력(인터페이스·모의 데이터):
내보낼 결과(함수·이벤트·타입):
수정해도 되는 영역:
절대 하면 안 되는 것:
관련 미결정 ID:
성공 테스트:
실패 테스트:
다른 트랙에 인계할 항목:
```

AI 코딩 에이전트에는 최소한 다음 조건을 함께 준다.

```text
AGENTS.md와 지정된 docs를 먼저 읽어라.
이번 작업의 AC 범위만 구현하라.
docs/11-interfaces.md의 의미를 임의로 바꾸지 마라.
13-open-decisions.md 항목은 상수로 확정하지 말고 TODO(decision): ID를 남겨라.
성공 경로와 실패 경로 테스트 이름에 AC ID를 넣어라.
모의 구현 파일 첫 줄에 // MOCK: 가정을 적어라.
완료 후 변경 파일, 실행 명령, 테스트 결과, 다른 트랙에 주는 인터페이스를 보고하라.
```

### 4.4 한 작업의 완료 조건

코드만 있으면 완료가 아니다. 아래 다섯 가지가 함께 있어야 한다.

1. 명세와 일치하는 구현
2. AC 번호가 붙은 성공·실패 테스트
3. 다른 트랙이 호출할 공개 인터페이스와 타입
4. 로컬 실행 방법과 필요한 설정값 예시
5. 모의 가정, 미결정 사항, 알려진 제한의 기록

## 5. 트랙 ① 규제 — 신원과 클레임

### 이 팀의 목표

개인정보를 공개하지 않으면서 “이 지갑은 필요한 검증을 통과했다”는 사실만 다른 컨트랙트가 확인할 수 있게 한다.
법을 새로 해석하는 팀이 아니라, 승인된 판단 결과를 안전하게 표현하고 검증하는 팀이다.

### 먼저 볼 문서

- [01-regulation.md](./01-regulation.md)
- [10-domain-model.md](./10-domain-model.md) §1, §2 Investor, §4
- [11-interfaces.md](./11-interfaces.md) §3
- [12-acceptance.md](./12-acceptance.md) AC-1, AC-8.2~8.3
- [13-open-decisions.md](./13-open-decisions.md) A, D-C6

### 구현 순서

1. `IIdentityRegistry`에 맞춰 필수 클레임의 존재와 신뢰 가능한 발급자 서명을 검증한다.
2. 정상, 누락, 철회된 클레임을 각각 재현할 테스트 도구를 만든다.
3. Alice와 Bob의 가짜 지갑·ONCHAINID fixture를 만든다. 이름은 시나리오용 별칭일 뿐 개인정보를 넣지 않는다.
4. 토큰화 팀이 같은 fixture를 가져다 쓸 수 있게 검증 함수와 설정 방법을 문서화한다.
5. AC-1 전체를 통과시키고, 배당 팀이 조세 거주성 결과를 안전하게 소비할 경계를 확인한다.

### 비개발 팀원이 확인할 것

- 어느 클레임이 필수인지와 누가 발급할 수 있는지 표로 검수한다.
- 여권번호, 이름, 주소, 생년월일과 그 해시가 온체인 입력·로그·fixture에 없는지 확인한다.
- 법률 쟁점이 나오면 답을 만들지 말고 D-A 항목으로 올린다.

### 다른 팀에 넘길 것

- `IIdentityRegistry` 구현의 ABI·주소·설정 방법
- 신뢰 발급자와 클레임 topic을 주입하는 방법
- 정상 Alice, 정상 Bob, 클레임 누락, 철회, 만료 fixture
- 검증 실패를 구분하는 오류 또는 결과

### 금지선과 보류선

- 개인정보 원본이나 해시를 온체인에 넣지 않는다.
- 컨트랙트가 투자자 적격성을 스스로 판단하게 만들지 않는다.
- 클레임 topic 번호는 D-C6 결정 전 임의로 고정하지 않는다.
- 조세 거주성 표현 방식과 세율 데이터 경계는 D-B8 결정 전 임의로 확정하지 않는다.

## 6. 트랙 ② 인프라 — 오라클, 커스터디, 1차 DvP

### 이 팀의 목표

KRX·KSD·수탁기관·결제 원장을 직접 연결할 수 없는 PoC 환경에서, 외부 데이터를 안전하게 전달하고
성공뿐 아니라 지연·실패·복구까지 재현한다.

### 먼저 볼 문서

- [02-infrastructure.md](./02-infrastructure.md)
- [10-domain-model.md](./10-domain-model.md) §1, §3.4, §4
- [11-interfaces.md](./11-interfaces.md) §4, §6, §7
- [12-acceptance.md](./12-acceptance.md) AC-3, AC-5.5~5.7, AC-6.6, AC-7
- [13-open-decisions.md](./13-open-decisions.md) B, C, D

### 구현 순서

1. 테스트가 값과 `updatedAt`을 주입할 수 있는 외국인 한도·가격·기업행위 모의 피드를 만든다.
2. `maxAge`를 코드에 고정하지 않고 설정으로 받아 정상·스테일·누락 상태를 재현한다.
3. `CustodyAdapter`를 `settled | pending | failed` 세 결과로 동작하게 만들고 attestation 경로를 연결한다.
4. 1차 시장에 한해 `LOCK → VERIFY → FUND → COMMIT | ABORT` 오케스트레이터를 만든다.
5. 중복 이벤트와 재시도를 넣어도 같은 요청이 두 번 처리되지 않도록, 결정된 상관관계·멱등성 규약을 적용한다.
6. 각 모의 서비스의 시작 명령, 설정값, 실패 주입 방법을 적는다.

### 비개발 팀원이 확인할 것

- 실제 데이터의 권위자는 KRX·KSD·수탁기관이고 오라클은 전달자일 뿐인지 확인한다.
- 성공, 지연, 실패, 스테일, 타임아웃 시나리오가 모두 데모 가능한지 확인한다.
- ABORT 뒤 양쪽 자산이 원위치로 돌아오는지 결과를 확인한다.

### 다른 팀에 넘길 것

- `IForeignOwnershipOracle` ABI·주소와 정상/한도초과/스테일 fixture
- `CustodyAdapter` 타입, 실행 주소, 지연·실패 주입법
- `DvpOrchestrator` 타입과 COMMIT/ABORT 결과·로그
- 로컬 원장 접속 정보와 배포 결과물

### 금지선과 보류선

- 외국인 총보유 비율을 토큰 잔고 합으로 계산하지 않는다.
- 스테일 데이터를 “일단 통과”시키지 않는다.
- 2차 매매를 DvP 오케스트레이터나 대기 상태에 넣지 않는다.
- 로컬 원장 구성(D-B4), `maxAge`·갱신주기·마진(D-C1~C3), Finality·deadline(D-C5)을 임의로 고정하지 않는다.
- 예상 보유비율 산식의 외부 입력은 D-B7, 1차 요청 상관관계는 D-B9 결정에 따른다.

## 7. 트랙 ③ 토큰화 — 주식 토큰과 생애주기

### 이 팀의 목표

온체인 잔고가 곧 법적 주식 수량이라는 전제 아래, 보유·이전·발행·배당·환매를 구현하고 모든 경로에서
규제와 수탁 불변식을 강제한다.

### 먼저 볼 문서

- [03-tokenization.md](./03-tokenization.md)
- [10-domain-model.md](./10-domain-model.md) 전체
- [11-interfaces.md](./11-interfaces.md) §1, §2, §5
- [12-acceptance.md](./12-acceptance.md) AC-2~AC-6, AC-8
- [13-open-decisions.md](./13-open-decisions.md) B, C, D-E1

### 구현 순서

1. `IStockToken`과 신원 레지스트리를 연결한다.
2. 신원·자격, 락업, 적격 보유 주소, 제재, 외국인 한도 모듈을 각각 독립 구현한다.
3. 각 모듈이 자기 이름으로 구분되는 실패를 반환하는지 단위 테스트한다.
4. 2차 이전을 단일 트랜잭션으로 완결하고 중간 상태를 만들지 않는다.
5. mint에도 다섯 모듈을 적용해 ERC-3643 기본 mint 갭을 막는다.
6. 1차 발행과 환매 상태기계, 실패·환불·잠금 해제 경로를 구현한다.
7. 기업행위 피드와 조세 거주성 결과를 사용해 배당을 투자자별로 분리 계산한다.

### 비개발 팀원이 확인할 것

- Alice 100주 매수, Bob에게 40주 매도 뒤 잔고가 Alice 60 / Bob 40인지 확인한다.
- 2차 거래에는 Pending이 없고, 1차 발행·환매에만 Pending이 있는지 확인한다.
- 발행 전 실물 확보 attestation이 없으면 토큰이 생기지 않는지 확인한다.
- 다섯 모듈 각각에 “이 모듈 때문에 거절”되는 테스트가 있는지 확인한다.

### 다른 팀에 넘길 것

- `IStockToken`, `IPrimaryMarketVault`, 컴플라이언스 모듈의 ABI·배포 주소
- 이벤트와 custom error 목록 및 각 필드의 의미
- Alice/Bob 잔고와 발행·환매 요청을 만드는 테스트 도구
- 에이전트가 사전 확인·실행·오류 설명에 사용할 호출 예시

### 금지선과 보류선

- 토큰을 청구권, 래핑 자산, 예탁증서로 모델링하지 않는다.
- 관리자 편의를 위한 컴플라이언스 우회 함수를 만들지 않는다.
- 2차 경로에 request/claim을 넣지 않는다.
- 배당 세율을 하드코딩하지 않는다.
- 2차 주문·대금 경계(D-B6), 예상 보유비율 산식(D-B7), 배당 세율 경계(D-B8)를 결정 전 임의 구현하지 않는다.

## 8. 트랙 ④ 에이전트 — 자연어 주문 UX

### 이 팀의 목표

사용자의 자연어를 구조화된 주문으로 바꾸되, 모호한 값은 되묻고 위임 범위를 넘으면 사람 확인을 받으며,
컨트랙트의 성공과 거절을 사실대로 설명한다.

### 먼저 볼 문서

- [04-agent-ux.md](./04-agent-ux.md)
- [10-domain-model.md](./10-domain-model.md) §1, §2 Mandate
- [11-interfaces.md](./11-interfaces.md) §8와 호출할 §2, §4
- [12-acceptance.md](./12-acceptance.md) AC-9, AC-10.4
- [13-open-decisions.md](./13-open-decisions.md) D-C7, D-B6

### 구현 순서

1. `OrderIntent` 입력 검증과 종목명→종목코드 정규화를 먼저 만든다.
2. Intent Parser가 누락·모호한 슬롯을 추측하지 않고 `needs_clarification`으로 반환하게 한다.
3. Policy Guard가 Mandate 안/밖과 사람 확인 필요 여부를 분리한다.
4. Order Builder가 오라클 가격으로 금액을 수량으로 바꾸고 한도가를 반영한다.
5. Executor가 토큰화 팀의 공개 인터페이스만 호출하게 한다.
6. Explainer가 custom error를 숨기지 않고 안전한 사용자 문장으로 매핑한다.
7. LLM과 체인 연결은 교체 가능한 어댑터 뒤에 두고, 테스트에서는 결정론적 가짜 구현을 사용한다.

### 비개발 팀원이 확인할 것

- 실제 사용자가 할 법한 정상·모호·위임 초과·규제 거절 문장을 준비한다.
- “적당히” 같은 말을 임의 수량으로 바꾸지 않는지 확인한다.
- 제재 사유처럼 노출을 줄여야 하는 정보와, 갱신 방법을 알려줘야 하는 오류를 구분해 문구를 검수한다.
- 거절된 주문을 성공으로 말하거나 쪼개서 우회하지 않는지 확인한다.

### 다른 팀에 넘길 것

- `TradingAgent.handle()` 입력·출력 타입과 호출 예시
- 정상 실행, 되묻기, 사람 확인, 컴플라이언스 거절 fixture
- 지원하는 발화 목록과 아직 지원하지 않는 표현
- 실행·오류 추적에 필요한 로그와 데모 스크립트

### 금지선과 보류선

- 에이전트가 KYC·제재·외국인 한도를 자체 판정하지 않는다.
- 누락된 종목, 금액, 수량을 추측하지 않는다.
- 거절을 피하려고 주문을 자동 분할하거나 다른 경로로 재시도하지 않는다.
- 기본 Mandate 값은 D-C7 결정 전 하드코딩하지 않는다.
- 실제 매수 실행은 D-B6에서 정한 2차 주문·대금 인터페이스를 사용한다.

## 9. 트랙 사이의 연결 계약

[11-interfaces.md](./11-interfaces.md)가 공개 계약의 기준이다. 다른 팀의 내부 파일을 직접 가져다 쓰지 말고,
아래 공개 결과물만 소비한다.

| 제공 팀 | 반드시 제공할 것 | 주 소비 팀 | 합치기 전 확인 |
|---------|------------------|------------|----------------|
| 규제 | 신원 레지스트리 ABI·주소, 클레임 fixture | 토큰화, 에이전트 | 정상·누락·철회 결과가 재현됨 |
| 인프라 | 오라클 ABI·주소, 커스터디/DvP 타입과 모의 제어 | 토큰화, 에이전트 | 성공·스테일·지연·실패·ABORT가 재현됨 |
| 토큰화 | 토큰·볼트 ABI·주소, 이벤트·custom error | 인프라, 에이전트 | mint 포함 규제 검사와 상태 전이가 관측됨 |
| 에이전트 | 구조화 의도와 네 종류의 `AgentReply` | 통합 테스트 | 실행·거절·확인·되묻기가 구분됨 |

공개 인터페이스를 바꿔야 할 때는 다음을 한 PR에서 함께 한다.

1. [11-interfaces.md](./11-interfaces.md)의 의미와 시그니처를 먼저 갱신한다.
2. 제공 팀 구현과 테스트를 갱신한다.
3. 소비 팀의 mock·타입·호출부를 갱신한다.
4. 변경된 AC와 INV 테스트를 실행한다.
5. PR 설명에 호환성 영향과 필요한 후속 작업을 적는다.

추가 공통 규칙은 다음과 같다.

- 주소, RPC URL, 임계값은 코드에 박지 않고 설정으로 주입한다.
- ABI와 생성 타입은 실제 배포 코드에서 다시 만들 수 있어야 한다.
- 이벤트 이름과 custom error 필드는 사용자 설명과 감사 로그가 의존하므로 조용히 바꾸지 않는다.
- 모든 모의 구현은 파일 상단에 `// MOCK:`과 실제 연계 시 달라지는 점을 적는다.
- 테스트 데이터에는 실제 개인정보나 실제 계정 비밀키를 넣지 않는다.

## 11. PR과 인계 체크리스트

저장소의 기존 협업 원칙대로 `main`에 직접 push하지 않고 `rwa/<짧은-작업명>` 브랜치에서 PR을 만든다.

PR을 열기 전에 다음을 확인한다.

- [ ] 작업 카드에 책임 AC가 적혀 있다.
- [ ] 테스트 이름에 `AC-x.y`가 들어 있다.
- [ ] 성공과 실패 경로가 모두 테스트됐다.
- [ ] 관련 INV가 깨지지 않는다.
- [ ] 공개 인터페이스 변경 시 11 문서와 소비 팀 mock도 함께 바뀌었다.
- [ ] 미결정 값을 임의로 정하지 않았고 필요한 `TODO(decision): D-X0`가 있다.
- [ ] 모의 구현에 `// MOCK:` 가정이 있다.
- [ ] 실제 개인정보·비밀키·토큰이 없다.
- [ ] 실행 명령, 설정값 예시, 예상 결과가 적혀 있다.
- [ ] 다른 트랙이 받아야 할 ABI·타입·주소·fixture가 명시돼 있다.

권장 PR 설명 양식:

```text
책임 트랙 / AC:
무엇이 이제 동작하는가:
실행 방법:
테스트 결과:
실패 시나리오:
공개 인터페이스 변경:
모의 가정:
관련 결정 / TODO:
다른 트랙의 후속 작업:
```

## 13. 첫 킥오프에서 끝낼 일

첫 회의의 목표는 긴 설계 토론이 아니라 다음 여섯 가지를 확정하는 것이다.

1. 트랙별 도메인 담당자와 구현 담당자
2. 통합 담당자
3. 각 트랙의 첫 AC 작업 한 개
4. D-B4, D-B6~D-B9의 결정 담당자와 기한
5. 공통 fixture와 인터페이스 PR의 담당자
6. 첫 통합 데모 날짜와 통과할 AC 범위

이 여섯 가지가 정해지면 각 팀은 독립적으로 시작하되, 동일한 인터페이스와 검수표를 향해 작업할 수 있다.
