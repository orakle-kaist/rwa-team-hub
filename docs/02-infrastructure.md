# 02 — 인프라 (네트워크 · 브릿지 · 오라클)

---

## 1. 왜 원장을 둘로 나누는가

증권 결제의 기본 원칙은 **DvP(Delivery versus Payment)** — 자산 이전과 대금 지급이 **동시에** 완결돼야
한쪽만 이행되고 다른 쪽이 펑크 나는 위험이 사라진다.

문제는 자산과 대금의 요구 조건이 정반대라는 것이다.

| | RWA 자산 원장 | 결제 대금 원장 |
|---|---------------|----------------|
| 필요한 것 | 퍼블릭 유동성, 생태계 접근성 | 되돌릴 수 없는 즉시 완결성, 프라이버시 |
| 기본 선택 | OP Stack 계열 L2 | Hyperledger Besu (QBFT) |
| Finality | 경제적 완결성은 수초, **법적 완결성 7일** | 1~3초 BFT 즉시 완결 |
| 근거 | ERC-3643 등 EVM 표준 재사용, 이더리움 유동성 | 결제가 확정 후 되돌려지면 결제 기능 자체가 무너짐 |

단일 원장으로 몰면 매 거래마다 현금을 미리 쌓아야 하고(유동성 부담), 상계결제 이점이 사라진다.

> **주의:** OP Stack의 법적 Finality 7일은 토큰 표준으로 해소되지 않는 구조적 특성이다.
> 그래서 자산 유통 레이어만 담당시키고, 대금 결제는 Besu가 맡는다.

> **단일 체인으로 부족한 이유:** 결제를 퍼블릭 L2 하나에서만 처리하려면 현금을 미리 온체인에 예치해야 하고(유동성 부담), 건별 자금 이동이 필요해 상계결제 이점이 사라진다. 반대로 자산도 허가형 체인에서만 관리하면 ERC-3643·ERC-7540 같은 공개 표준과 이더리움 생태계 도구를 활용하기 어렵다.

## 2. 브릿지와 오케스트레이터

### 오케스트레이터란 무엇인가

두 원장은 서로를 직접 볼 수 없다. **오케스트레이터**는 그 사이에서 순서를 맞추는 실행 계층이다.
자산을 잠그고, 대금을 확보하고, 컴플라이언스를 확인한 뒤, 양쪽이 다 준비됐을 때만 동시에 실행하고
아니면 되돌린다.

실제 사례로 Chainlink의 **CRE(Chainlink Runtime Environment)** 가 이 역할을 하며,
J.P. Morgan Kinexys와 Ondo Finance의 크로스체인 DvP에서 사용됐다.

### 조건부 DvP 흐름

```
LOCK    자산 원장에서 주식 토큰을 에스크로에 잠근다
  ↓
VERIFY  브릿지(CCIP)가 잠금 상태를 결제 원장에 전달·검증한다
  ↓
FUND    결제 원장에서 원화 예금토큰(대금)을 확보한다
  ↓
COMMIT  양쪽 조건이 모두 충족 → 동시에 실행 (증권 이전 + 대금 지급)
  또는
ABORT   시간 초과 또는 조건 미충족 → 양쪽 에스크로 해제
```

> **ABORT가 핵심 안전장치다.** "한쪽만 넘어간" 상태를 남기지 않는 것이 이 흐름의 목적이다.
> 실패는 예외가 아니라 정상 경로다. 반드시 구현하고 테스트한다.

### 크로스체인 DvP가 풀어야 할 세 문제

| 문제 | 내용 | 대응 |
|------|------|------|
| 최종성(Finality) | 자산 원장의 경제적 완결성(수초)과 법적 완결성(7일)이 다름 | 오케스트레이터가 Finality 정책을 적용해 commit 시점 결정 |
| 자산 동일성 | 체인마다 같은 이름의 토큰이 생길 수 있음 | 종목코드·ISIN ↔ 체인별 토큰 주소를 잇는 자산 등록부 |
| 결제 동시성 | 한쪽만 이전되는 상황 방지 | 양쪽 에스크로 잠금 후 2단계 커밋 |

> **적용 범위 주의:** 이 DvP 흐름은 **1차 시장(발행·환매)** 에 적용된다.
> 2차 매매는 단일 자산 원장 안에서 동기적으로 끝나므로 이 흐름을 타지 않는다. → `AGENTS.md` R4

## 3. 오라클

블록체인은 구조적으로 닫힌 시스템이라 스마트컨트랙트가 외부 데이터를 스스로 가져올 수 없다.
토큰화 증권에서는 **세 축**이 모두 필요하다.

| 축 | 데이터 | 권위 있는 원천 | 갱신 주기 |
|----|--------|----------------|-----------|
| **가치** | 종목 시세, NAV, 담보 준비금 증명(PoR) | KRX, 수탁기관 | 장중 실시간 / 일 1회 |
| **권리** | 배당락·배당률, 주식분할 등 기업행위 | KRX 전자공시, KSD | 이벤트 기반 |
| **규제** | **외국인 종목별 한도 및 현재 총보유 비율**, 제재 목록 | KRX, KSD | 일 1회 이상 |

세 축 중 하나라도 빠지면 자산의 껍데기만 올린 셈이 된다.

### 핵심 원칙: 권위(Authority)와 전달(Delivery)의 분리

> 데이터의 **권위**는 KRX·KSD 같은 법적 원천 기관에 있고, 오라클은 **전달자**일 뿐이다.
> 오라클은 값의 생산자가 아니다.

### 외국인 한도 데이터 (가장 중요)

오라클이 전달해야 할 값은 두 가지다.

1. **종목별 법정 한도** — 법령 개정 시에만 변함 (예: 전기통신사업 49%)
2. **현재 외국인 총보유 비율** — 장중 지속 변동, **장내·장외 전체 집계값**

컨트랙트는 자신이 관리하는 토큰의 보유 현황만 알 뿐, 같은 종목의 오프체인 보유분을 알 수 없다.
따라서 이 값은 **반드시 외부에서 공급받아야 한다.** → `AGENTS.md` R5

### 스테일 데이터 처리 (필수 구현)

한도 데이터가 오래됐을 때의 동작을 반드시 정의해야 한다.

- 피드에 `updatedAt` 타임스탬프를 포함한다
- 허용 지연(staleness threshold)을 넘으면 **보수적으로 거절**한다 (통과시키지 않는다)
- 갱신 주기를 길게 가져갈 경우 보수적 마진을 두거나, 한도 근접 종목은 이원화한다
- 구체적 임계값은 [13-open-decisions.md](./13-open-decisions.md) 참조

## 4. 로컬 개발 환경에서의 대체

PoC 단계에서는 실제 인프라에 접속할 수 없으므로 아래로 대체한다.
모든 모의 구현은 파일 상단에 `// MOCK:` 주석으로 가정을 명시한다.

| 실제 | 로컬 대체 |
|------|-----------|
| OP Stack L2 | Anvil 인스턴스 #1 |
| Hyperledger Besu | Anvil 인스턴스 #2 (또는 동일 체인 내 분리된 컨트랙트) |
| Chainlink CCIP | 모의 릴레이 서비스 (이벤트 감시 → 반대편 호출) |
| Chainlink 가격/한도 피드 | 모의 오라클 컨트랙트 (테스트에서 값 주입 가능) |
| KSD 대외 연계 | 타이머 기반 모의 어댑터 (T+2 시뮬레이션) |
| 수탁기관 실물 매매 | 모의 API (즉시 성공 / 지연 / 실패 시나리오 주입 가능) |

> 모의 어댑터는 **성공 경로만 구현하지 말 것.** 지연·실패·부분 체결 시나리오를 주입할 수 있어야
> [12-acceptance.md](./12-acceptance.md)의 실패 경로 테스트가 가능하다.
