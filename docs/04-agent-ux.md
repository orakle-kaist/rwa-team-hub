# 04 — AI 에이전트 (자연어 거래 UX)

---

## 1. 이 트랙이 푸는 문제

앞 트랙까지 오면 진짜 주식이 온체인에서 즉시 거래된다.
그런데 일반 투자자에게 지갑 주소·주문 수량·가스는 여전히 장벽이다.

**목표:** 사용자가 평소 말로 지시하면, 에이전트가 그 의도를 규칙에 맞는 온체인 주문으로 바꿔 실행한다.

```
사용자: "하이닉스 500만 원어치 사줘. 너무 비싸지만 않으면."
   ↓
에이전트: 의도 파싱 → { 종목: SK하이닉스, 방향: 매수, 금액: 5,000,000원, 한도가: 현재가 이내 }
   ↓
        신원·한도 자동 확인 → 2차 시장 즉시 체결
   ↓
에이전트: "SK하이닉스 26주를 500만 원에 매수했어요. 신원·외국인 한도 확인 후 즉시 체결됐습니다."
```

---

## 2. 핵심 원칙: 말은 자유롭게, 규칙은 그대로

| 말은 자유롭게 | 규칙은 그대로 |
|---------------|---------------|
| 지갑 주소·주문 수량을 몰라도 됨 | 모든 주문이 신원·외국인 한도 검사를 거침 |
| "비싸지 않으면", "천천히 나눠서" 같은 조건도 이해 | 통과 못 하면 **에이전트도 실행 불가** (거절) |
| 매수·매도·잔고 조회를 한 대화로 | 위임 범위·한도를 넘는 요청은 **사람 확인** |

> **자연어라고 규제가 느슨해지지 않는다.**
> 이게 가능한 이유는 규칙이 토큰 안에 박혀 있어서, 인터페이스가 말로 바뀌어도 통제 지점이 그대로이기 때문이다.
> 에이전트는 규칙을 *해석*하지 않는다. 주문을 넣을 뿐이고, 판정은 컨트랙트가 한다.

---

## 3. 아키텍처

```
사용자 발화
   │
   ▼
[Intent Parser]        LLM tool-calling. 자연어 → 구조화된 주문 의도
   │                   미확정 슬롯이 있으면 되묻는다 (임의로 채우지 않음)
   ▼
[Policy Guard]         위임 범위(mandate) 검사: 금액 한도, 종목군, 만료, 방향
   │                   범위 초과 → 사람 확인 요청으로 분기
   ▼
[Order Builder]        금액 → 수량 환산 (오라클 시세 사용), 슬리피지·한도가 반영
   │
   ▼
[Executor]             온체인 트랜잭션 제출. 결과/실패 사유 수신
   │
   ▼
[Explainer]            체결 결과 또는 거절 사유를 사용자 언어로 설명
```

### 각 계층의 책임 경계

| 계층 | 해도 되는 것 | 하면 안 되는 것 |
|------|--------------|------------------|
| Intent Parser | 발화에서 슬롯 추출, 모호하면 되묻기 | 누락된 값을 추측해서 채우기 |
| Policy Guard | 위임 범위 초과 여부 판단 | 규제 적격성 판단 (그건 컨트랙트 몫) |
| Order Builder | 시세 기반 수량 환산 | 한도 검사 결과 예단 |
| Executor | 트랜잭션 제출·재시도 | 실패를 성공으로 보고하기 |
| Explainer | 사실 그대로 설명 | 거절 사유를 얼버무리기 |

## 4. 주문 의도 스키마

```typescript
type OrderIntent = {
  action: "buy" | "sell" | "balance" | "dividend_history";
  symbol: string;              // 종목 (예: "SK하이닉스" → 종목코드로 정규화)
  // 금액 기준 또는 수량 기준 중 하나
  notional?: { currency: "KRW" | "USDC"; amount: string };
  quantity?: string;           // 주 단위
  limitPrice?: string;         // "너무 비싸지만 않으면" → 현재가 기준 상한
  timeInForce?: "immediate" | "day";
};
```

**규칙**
- `notional`과 `quantity`는 배타적이다. 둘 다 없으면 되묻는다.
- 종목명은 반드시 **종목코드로 정규화**한 뒤 실행한다. 동음이의·유사명은 사용자에게 확인한다.
- 모호한 표현("적당히", "좀")은 추측하지 말고 되묻는다.

## 5. 위임(Mandate) 모델

에이전트는 사용자가 부여한 범위 안에서만 자율 실행한다.

```typescript
type Mandate = {
  maxNotionalPerOrder: string;   // 건별 상한
  maxNotionalPerDay: string;     // 일별 누적 상한
  allowedSymbols?: string[];     // 미지정 시 전체 허용
  allowedActions: ("buy" | "sell")[];
  expiresAt: string;
  requireHumanApprovalAbove: string;  // 이 금액 초과 시 사람 확인
};
```

- 범위를 넘는 요청은 **거절이 아니라 사람 확인으로 분기**한다.
- 위임 범위는 규제 적격성과 **별개**다. 위임 안이어도 컨트랙트 검사는 그대로 적용된다.

---

## 6. 거절 처리 (중요)

컨트랙트에서 거절되는 것은 **정상 동작**이다. 에이전트는 이를 숨기거나 우회하지 않는다.

| 거절 사유 | 사용자에게 설명할 내용 |
|-----------|------------------------|
| 신원 클레임 없음/만료 | 어떤 검증이 필요한지, 어디서 갱신하는지 |
| 외국인 한도 초과 | 해당 종목이 한도에 도달했음. 수량을 줄이면 가능한지 |
| 제재 스크리닝 | 거래를 진행할 수 없음 (상세 사유는 노출 최소화) |
| 락업 | 해제 예정 시점 |
| 오라클 데이터 스테일 | 일시적으로 확인 불가, 재시도 안내 |

> 에이전트가 거절을 우회하려고 주문을 쪼개거나 다른 경로를 시도하는 동작을 구현하지 말 것.

---

## 7. 구현 체크리스트

- [ ] Intent Parser가 누락 슬롯을 추측하지 않고 되묻는다
- [ ] 종목명이 종목코드로 정규화된 뒤 실행된다
- [ ] Mandate 초과 요청이 사람 확인으로 분기된다
- [ ] 에이전트가 규제 판정을 대신하지 않는다 (컨트랙트 결과를 그대로 수용)
- [ ] 거절 사유가 사용자 언어로 정확히 설명된다
- [ ] 실패를 성공으로 보고하는 경로가 없다
- [ ] 거절 우회(주문 쪼개기 등) 로직이 없다
