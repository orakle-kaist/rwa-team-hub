## 1. 외국인이 한국 주식을 사려면 (충족 요건)

세 가지를 **모두** 통과해야 보유·이전할 수 있다.

| # | 요건 | 근거 | 성격 |
|---|------|------|------|
| 1 | **신원 확인 · 계좌 개설** — 여권번호(개인)/LEI(법인) 식별 + 고객확인(KYC) | 특정금융정보법 | 개인 단위 판정 |
| 2 | **외국환 신고 · 환전** — 자금 유출입 신고, 원화 환전 | 외국환거래법 | 개인 단위, 오프체인 완결 |
| 3 | **외국인 지분 한도** — 종목별 법정 상한 | 전기통신사업법 §8, 방송법 §14 등 | **종목 전체 집계값** |

### 주의사항 (구현 시 자주 틀리는 부분)

- **VASP 규제는 해당하지 않는다.** VASP(가상자산사업자) 고객확인은 디지털자산 거래에 적용되는 규제다.
  한국 상장주식 매매는 일반 증권계좌의 고객확인 절차를 따른다. 두 개를 섞지 말 것.
- **외국인투자자 등록제(IRC)는 2023년 12월 폐지**됐다. 이제 개인은 여권번호, 법인은 LEI로 식별한다.
  "FSS 사전등록" 같은 단계를 구현에 넣지 말 것.
- 1·2번은 개인 단위로 판정되지만 **3번은 성격이 다르다.** 같은 사람의 같은 주문이라도
  그 시점의 종목별 외국인 총보유 비율에 따라 허용되기도, 거부되기도 한다.

## 2. 판단(Decision)과 집행(Enforcement)의 분리

규제 준수를 두 층위로 나눈다. **이 분리가 이 설계의 기반이다.**

| 구분 | 판단 (Decision) | 집행 (Enforcement) |
|------|-----------------|---------------------|
| 하는 일 | 규제 요건 충족 여부를 결정 | 결정된 요건을 거래에 적용 |
| 입력 | 개인정보, 증빙 서류, 외부 데이터 | 온체인 상태값, 서명된 클레임 |
| 성격 | 해석과 재량이 개입 | 결정론적 반복 |
| 예외 처리 | 가능 (행정 절차·재심사) | 불가 (코드에 없는 예외는 실행 불가) |
| 책임 주체 | 금융기관, 인증기관, 규제기관 | 컨트랙트 배포·운영 주체 |
| 위치 | **오프체인** | **온체인** |

> **구현 규칙:** 온체인으로 넘어가는 것은 *판단의 결과*(검증 기관이 서명한 클레임)뿐이다.
> 판단의 *근거*(개인정보·심사자료)는 오프체인에 남는다. → `AGENTS.md` R1, R2

> **비개발자 읽기:** 은행이 "이 사람은 고객확인을 통과했다"고 도장 찍는 것이 **판단**이고,
> 컨트랙트가 "도장 있으면 이전 허용"을 실행하는 것이 **집행**이다.
> 도장 찍는 논리(심사 기준·재량)는 코드 안에 없다. 온체인으로 넘어가는 것은 *도장 자체*(클레임 서명)뿐이다.

### 왜 전부 온체인에서 할 수 없는가

1. **개인정보는 파기할 수 없는 곳에 기록할 수 없다.** 개인정보보호법 §21은 보유기간 경과 시 지체 없는
   파기를 요구하는데, 블록체인은 사후 삭제가 불가능하다. 여권번호는 저엔트로피 값이라 해시해도
   전수 대조로 원본 역산이 가능하다.
2. **법적 권위는 코드가 만들 수 없다.** 권리 이전의 효력은 전자등록계좌부에서 발생한다.
3. **총량 규제는 프로토콜 경계 밖 정보를 요구한다.** 외국인 총보유 비율은 시장 전체 집계값이라
   컨트랙트가 자기 잔고를 합산해서 만들 수 없다. → `AGENTS.md` R5
4. **규제에는 예외 처리와 행정적 판단이 필요하다.** 법원 가처분, 금융사고 대응 등.

## 3. 발행인계좌관리기관 구조 (핵심 결론)

### 기존 구조

```
발행기관
   │
명의개서대리인
   │
KSD (한국예탁결제원)
   │
일반계좌관리기관 (= 증권사)   ← 투자자별 계좌부를 여기서 관리
   │
투자자
```

증권사가 투자자별 계좌부를 관리하므로 온체인이 끼어들 자리가 없다.

### 토큰증권 구조

```
발행기관
   │
(명의개서대리인 — 없거나 역할 축소)
   │
KSD (한국예탁결제원)
   │
발행인계좌관리기관             ← 발행회사가 직접. 계좌부 = 온체인 분산원장
   │
투자자
```

개정 전자증권법상 요건을 갖춘 발행인은 **발행인계좌관리기관**으로 등록해
자신이 발행한 주식의 투자자별 계좌부를 분산원장으로 관리할 수 있다.

### 이것이 바꾸는 것

- 바뀌는 칸은 **하나뿐**이다 (증권사 → 발행회사). 나머지 참여자 구조는 유지된다.
- 그 온체인 원장이 **법적 효력을 갖는 전자등록계좌부**가 된다.
- 따라서 토큰은 청구권이 아니라 **전자등록주식 그 자체**이고, 온체인 이전이 곧 주식 양도다.
- 온체인 잔고와 법적 소유가 어긋나는 순간이 존재하지 않는다.

> **구현 규칙:** 청구권·래핑·예탁증서 모델로 되돌아가는 구현을 하지 말 것. → `AGENTS.md` R3

## 4. 온체인에 올라가는 것 / 올라가지 않는 것

| 항목 | 온체인 | 형태 |
|------|--------|------|
| KYC 통과 여부 | ✅ | ClaimIssuer 서명 클레임 (topic: KYC) |
| 투자자 적격성 | ✅ | 클레임 |
| 관할(거주국) | ✅ | 클레임 |
| 조세 거주성 | ✅ | 클레임 (배당 세율 적용에 사용) |
| 외국환 신고 완료 여부 | ✅ | 클레임 |
| 승인된 지갑 목록 | ✅ | 화이트리스트 상태 |
| 외국인 총보유 비율 | ✅ | **오라클 공급값** (온체인 산출 불가) |
| 여권번호·이름·주소 | ❌ | 오프체인 (해시도 금지) |
| 심사 서류·증빙 | ❌ | 오프체인 |
| 자금세탁 위험 평가 근거 | ❌ | 오프체인 (결과만 클레임) |

## 5. 구현 체크리스트

- [ ] 개인정보가 온체인에 직·간접으로 기록되는 경로가 없다
- [ ] 컨트랙트에 적격성·제재·관할을 *판정*하는 로직이 없다 (결과 수용만)
- [ ] 외국인 한도 검사가 오라클 공급값을 사용한다 (자체 합산 아님)
- [ ] 토큰이 청구권이 아니라 주식 자체로 모델링돼 있다
- [ ] 발행(mint) 경로에도 컴플라이언스 검사가 적용된다
