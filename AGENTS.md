# AGENTS.md — 구현 에이전트 작업 규칙

이 레포에서 코드를 작성하는 모든 에이전트(및 사람)는 이 규칙을 따른다.
`CLAUDE.md`, `.cursorrules` 등이 필요하면 이 파일을 심볼릭 링크하거나 그대로 복사한다.

---

## 0. 시작 전 필수 절차

1. `docs/00-cycle-overview.md`를 읽어 전체 그림을 파악한다.
2. 지금 만들려는 것과 관련된 트랙 문서(01~04)를 읽는다.
3. `docs/10-domain-model.md`의 **불변식(Invariants)** 을 읽는다. 이건 협상 대상이 아니다.
4. `docs/12-acceptance.md`에서 이번 작업이 어떤 시나리오를 통과시켜야 하는지 확인한다.
5. `docs/13-open-decisions.md`를 확인한다. **여기 있는 항목은 임의로 결정하지 않는다.**

## 1. 절대 규칙 (위반 시 그 작업은 폐기)

### R1. 개인정보를 온체인에 올리지 않는다
여권번호·이름·주소·생년월일·심사 서류는 어떤 형태로도 온체인에 기록하지 않는다.
해시도 안 된다(여권번호는 저엔트로피라 전수 대조로 역산 가능).
온체인에는 **검증 기관이 서명한 클레임의 존재 여부**만 올린다.
단, 배당 계산에 필요한 **국가 코드**(조세 거주성)는 예외적으로 클레임 값에 허용한다.
이름·생년월일 등 직접 식별정보는 국가 코드여도 온체인에 넣지 않는다.

### R2. 컨트랙트는 규제를 "판단"하지 않는다
스마트컨트랙트는 오프체인에서 내려진 판단의 **결과**를 받아 집행만 한다.
적격성·제재·관할 판정 로직을 컨트랙트 안에 구현하지 말 것.
단, 온체인 목록(제재 리스트, 화이트리스트)에서 주소를 **조회**하는 것은 집행에 해당하므로 허용된다.
"판단"은 그 목록의 생성·해석 권한을 가리킨다.
→ 근거: `docs/01-regulation.md` §판단과 집행의 분리

### R3. 토큰 이전은 곧 주식 양도다
이 설계에서 토큰은 청구권이 아니라 전자등록주식 자체다.
"청구권", "래핑", "예탁증서" 같은 모델로 되돌리는 구현을 하지 말 것.
잔고와 법적 소유가 어긋나는 상태를 만들 수 있는 코드 경로는 금지.

### R4. 2차 매매에 대기 상태를 넣지 않는다
투자자 간 매매는 **동기적으로 즉시 완결**된다. 미리 수탁된 주식이 뒷받침하기 때문이다.
2차 경로에 ERC-7540 request/claim 패턴을 적용하지 말 것.
비동기 대기는 **1차(발행·환매)에만** 존재한다.

### R5. 총량 데이터는 온체인에서 만들지 않는다
외국인 보유 비율은 장내·장외를 합친 시장 전체 집계값이다.
컨트랙트가 자기 토큰 잔고를 합산해 이 값을 "계산"하면 틀린다. 반드시 오라클이 공급한 값을 쓴다.

### R6. mint 경로에도 컴플라이언스를 강제한다
ERC-3643 기본 구현의 `mint`는 `from == address(0)`이라 Modular Compliance 모듈을 건너뛴다.
이 레포에서는 발행 경로에도 동일한 검사를 강제한다(`moduleMintAction` 훅 또는 동등한 수단).
→ 이건 알려진 구조적 결함이다. `docs/03-tokenization.md` §mint 경로 참조.

### R7. 스펙 불명 시 24시간 결정, 결정 전 구현 금지
구현 중 인터페이스·행동 규격이 불분명한 경우, 임의로 가정하지 않는다.
`docs/13-open-decisions.md`에 항목을 등록하고 24시간 이내 팀 결정을 받는다.
결정 전에는 `TODO(decision): D-X0` 자리를 비워 두고 다음 AC로 이동한다.

## 2. 문서와 코드가 다를 때

| 상황 | 행동 |
|------|------|
| 문서가 틀렸다고 판단됨 | 코드를 먼저 고치지 말고 **문서 수정을 제안**하고 근거를 남긴다 |
| 문서에 없는 결정이 필요함 | `docs/13-open-decisions.md`에 항목을 추가하고, 잠정 선택을 `TODO(decision)`로 표시 |
| 문서끼리 모순됨 | 구현을 멈추고 모순을 보고한다. 임의로 한쪽을 고르지 않는다 |
| 법령 해석이 필요함 | **직접 해석하지 않는다.** 미결정 항목으로 올린다 |

## 3. 구현 순서 (권장)

의존성 때문에 아래 순서를 지키는 편이 안전하다.

```
1. identity/     ONCHAINID 클레임 검증 → 다른 모든 검사의 전제
2. token/        ERC-3643 + Modular Compliance 5개 모듈
3. oracle/       한도 피드 컨슈머 (모의 피드로 시작)
4. secondary     2차 매매 경로 (동기, 즉시) ← 여기까지가 핵심
5. primary/      ERC-7540 발행·환매 (비동기, 대기)
6. orchestrator/ 1차 DvP commit/abort
7. agent/        자연어 주문
```

각 단계는 `docs/12-acceptance.md`의 대응 시나리오가 통과해야 다음으로 넘어간다.

## 4. 테스트 규칙

- **인수 시나리오 = 테스트 이름.** `docs/12-acceptance.md`의 `AC-x.y` 식별자를 테스트 이름에 넣는다.
  예: `test_AC_2_1_secondaryBuySettlesInSameTx()`
- 실패 경로를 반드시 함께 테스트한다. 규제 컨트랙트에서 **거절되는 것이 정상 동작**이다.
- 각 컴플라이언스 모듈은 **어느 모듈에서 막혔는지 구분 가능한 revert reason**을 반환해야 한다.
  (감사·운영 대응 불가를 막기 위함. `docs/03-tokenization.md` §revert reason)
- 모의(mock) 구현에는 파일 상단에 `// MOCK:` 주석으로 무엇을 가정했는지 명시한다.

## 5. 코드 컨벤션

- 컨트랙트: Solidity, Foundry, OpenZeppelin 우선. 업그레이드가 필요하면 UUPS 프록시.
- 서비스: TypeScript, 명시적 타입. 상태 전이는 문자열 유니온이 아니라 열거형/판별 유니온으로.
- 주석은 **왜**를 적는다. 무엇을 하는지는 코드가 말한다. 특히 규제 근거는 문서 링크로 남긴다.
  ```solidity
  /// @dev 총보유 비율은 온체인에서 산출 불가 — 오라클 공급값 사용. docs/01-regulation.md §총량 규제
  ```
- 한국어 도메인 용어는 `docs/glossary.md`의 영문 식별자를 쓴다. (예: 발행인계좌관리기관 → `IssuerAccountManager`)

## 6. 하지 말 것

- 문서에 없는 새 액터/역할을 만들지 않는다 (예: "브로커 컨트랙트" 임의 추가)
- 편의를 위해 컴플라이언스 검사를 우회하는 관리자 함수를 만들지 않는다
- 실패를 조용히 삼키지 않는다. 실패는 명시적 상태(`failed`, `aborted`)로 남긴다
- 데모를 위해 실제 규제 요건을 생략하고 "나중에 추가" 주석만 남기지 않는다.
  생략이 필요하면 `docs/13-open-decisions.md`에 컴플라이언스 갭으로 등록한다
